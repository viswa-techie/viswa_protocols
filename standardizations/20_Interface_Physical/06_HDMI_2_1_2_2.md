**Topic:** HDMI 2.1 (48 Gbps); Fixed Rate Link (FRL); eARC; VRR; ALLM; QMS; DSC; HDMI 2.1a (SBTM); HDMI 2.2 (96 Gbps); FRL 12; 16K resolution

### 1.2 HDMI Signaling Evolution

| Era | Signaling | Mechanism | Bandwidth |
|:---:|:---------:|:---------:|:---------:|
| HDMI 1.0 - 2.0 | **TMDS** (Transition-Minimized Differential Signaling) | 3 data channels + 1 clock (10-bit encoding per channel) | Up to 18 Gbps |
| HDMI 2.1 | **FRL** (Fixed Rate Link) | 4 data lanes (no separate clock; embedded); 16b/18b encoding | Up to 48 Gbps |
| HDMI 2.2 | **FRL 12** (next-gen Fixed Rate Link) | 4 data lanes; improved encoding | Up to **96 Gbps** |

```mermaid
        PHY_H[Physical Layer<br/>━━━━━━━━━━━<br/>• FRL: 4 lanes × 12 Gbps = 48 Gbps<br/>  (HDMI 2.1)<br/>• FRL 12: 4 lanes × 24 Gbps = 96 Gbps<br/>  (HDMI 2.2)<br/>• 16b/18b encoding<br/>• AC-coupled differential]

    end

```

### 2.2 TMDS vs. FRL

| Property | TMDS (HDMI 1.0-2.0) | FRL (HDMI 2.1+) |
|:--------:|:---:|:---:|
| **Channels** | 3 data + 1 clock | **4 data (no separate clock)** |
| **Clock** | Separate TMDS clock channel (pixel clock × 10) | **Embedded** (clock recovered from data; like PCIe) |
| **Encoding** | 8b/10b (per channel) | **16b/18b** (per lane) |
| **Overhead** | 20% (8b/10b) | **11.1%** (16b/18b) |
| **Max per-channel/lane** | 6 Gbps (HDMI 2.0; TMDS char rate 600 MHz) | **12 Gbps** (HDMI 2.1); **24 Gbps** (HDMI 2.2) |
| **Max total** | 18 Gbps (3 × 6 Gbps) | **48 Gbps** (4 × 12); **96 Gbps** (4 × 24) |
| **FEC** | None | **RS(255,251)** Reed-Solomon per lane |
| **Backward compatible** | — | Yes: HDMI 2.1 devices fall back to TMDS for HDMI 1.x/2.0 sinks |

### 2.3 FRL Rate Levels

| FRL Level | Lanes | Per-Lane Rate | Total BW | Effective (after 16b/18b) | Typical Use |
|:---------:|:-----:|:---:|:---:|:---:|---|
| FRL 3L3 | 3 | 3 Gbps | 9 Gbps | 8 Gbps | 4K@30Hz (legacy compatibility) |
| FRL 3L6 | 3 | 6 Gbps | 18 Gbps | 16 Gbps | 4K@60Hz 8bpc |
| FRL 4L6 | 4 | 6 Gbps | 24 Gbps | 21.3 Gbps | 4K@60Hz 10bpc |
| FRL 4L8 | 4 | 8 Gbps | 32 Gbps | 28.4 Gbps | 4K@120Hz 8bpc |
| FRL 4L10 | 4 | 10 Gbps | 40 Gbps | 35.6 Gbps | 4K@120Hz 10bpc |
| **FRL 4L12** | **4** | **12 Gbps** | **48 Gbps** | **42.7 Gbps** | **8K@60Hz; 4K@120Hz HDR10** |
| FRL 4L24 (HDMI 2.2) | 4 | 24 Gbps | 96 Gbps | ~85 Gbps | 8K@120Hz; 16K@60Hz |

## Chapter 3 — HDMI 2.1 Key Features

### 3.1 Feature Matrix

| Feature | Description | Benefit | Who Uses |
|:-------:|-------------|---------|----------|
| **eARC** | Enhanced Audio Return Channel; up to 37 Mbps; supports Dolby Atmos / DTS:X object audio | Single HDMI cable: video TO TV + audio FROM TV to soundbar | TV + Soundbar |
| **VRR** | Variable Refresh Rate; display dynamically adjusts refresh to match source frame rate | No tearing; no stuttering in games | Gaming (PS5, Xbox, PC) |
| **ALLM** | Auto Low Latency Mode; TV automatically switches to game mode when game source detected | No manual mode switching; minimizes input lag | Gaming consoles + TVs |
| **QMS** | Quick Media Switching; seamless resolution/frame-rate changes without blackout | No blank screen when switching 24fps→60fps content | Streaming devices |
| **QFT** | Quick Frame Transport; reduces latency by sending frames in bursts (active data compressed in time) | Lower display latency for gaming/VR | Gaming; VR headsets |
| **DSC** | Display Stream Compression 1.2a; visually lossless 2:1-3:1 | Enables higher resolutions within 48 Gbps BW | 8K+; high refresh |
| **FEC** | Forward Error Correction; Reed-Solomon RS(255,251) per FRL lane | Correct bit errors in cable (essential at 12 Gbps/lane) | All FRL connections |
| **Dynamic HDR** | Frame-by-frame or scene-by-scene HDR metadata (HDR10+, Dolby Vision) | Optimal tone mapping per scene (vs. static HDR10 for whole movie) | HDR content |

### 3.2 eARC (Enhanced Audio Return Channel)

| Feature | ARC (HDMI 1.4) | eARC (HDMI 2.1) |
|:-------:|:---:|:---:|
| **Bandwidth** | ~1 Mbps | **37 Mbps** |
| **Audio formats** | Compressed: DD 5.1, DTS 5.1 | **Uncompressed: 7.1 PCM; Object: Dolby Atmos (TrueHD), DTS:X** |
| **Channels** | Up to 5.1 | **Up to 32 channels** |
| **Sample rate** | 48 kHz, 16-bit | Up to **192 kHz, 24-bit** (studio quality) |
| **Discovery** | CEC-based (unreliable) | **eARC Data Channel** (dedicated; reliable) |
| **Connection** | Shared with CEC pin (pin 14) | **Dedicated eARC differential pair** (pin 14 + Ethernet pins) |

### 3.3 Variable Refresh Rate (VRR)

```mermaid
sequenceDiagram
    participant GPU as GPU / Console
    participant TV as TV (VRR Sink)
    
    Note over GPU,TV: VRR enabled; TV accepts 48-120 Hz
    
    GPU->>TV: Frame 1 (rendered in 8.3ms) → send at 120 Hz
    GPU->>TV: Frame 2 (complex; rendered in 12ms) → send at 83 Hz
    GPU->>TV: Frame 3 (simple; rendered in 7ms) → send at 120 Hz (cap)
    GPU->>TV: Frame 4 (very complex; rendered in 20ms) → send at 50 Hz
    
    Note over GPU,TV: TV adjusts refresh to match EACH frame → no tearing, no judder
    Note over GPU,TV: Without VRR: fixed 60Hz → Frame 4 causes stutter or tear
```

**VRR vs. FreeSync vs. G-Sync vs. Adaptive-Sync:**

| Technology | Standard | Royalty | Interface |
|:----------:|:--------:|:-------:|:---------:|
| **HDMI VRR** | HDMI Forum spec | HDMI license fee | HDMI only |
| **AMD FreeSync** | Uses Adaptive-Sync (DP) or HDMI VRR | Free | DP + HDMI |
| **NVIDIA G-Sync** | Proprietary (hardware module) | NVIDIA license + module cost | DP (+ HDMI VRR compatible) |
| **VESA Adaptive-Sync** | VESA DP standard | Free (part of DP) | DisplayPort only |

### 4.2 Latency-Aware Transport (LAT)

| Aspect | Description |
|:------:|-------------|
| **Purpose** | Minimize end-to-end latency for gaming, VR, cloud gaming |
| **Mechanism** | Source and sink negotiate latency requirements; transport layer optimizes packet scheduling |
| **Target** | < 5 ms motion-to-photon (vs. current 15-30 ms for TV processing) |
| **Use case** | Cloud gaming (Stadia-like); VR headsets via HDMI; competitive gaming |

## Chapter 9 — Case Studies

### 9.1 Next-Gen Game Console HDMI 2.1 Implementation

| Aspect | Detail |
|--------|--------|
| **Product** | PS5 / Xbox Series X — first consoles with HDMI 2.1 |
| **Output modes** | 4K@120Hz (performance mode); 4K@60Hz HDR (quality mode); 8K@60Hz (future games) |
| **VRR** | Console tells TV: "this frame took 10ms" → TV refreshes at 100 Hz. Next frame: 16.7ms → 60 Hz. Prevents tearing without fixed vsync (which causes input lag). Range: 48-120 Hz (LFC doubles below 48). |
| **ALLM** | Console sends ALLM signal via HDMI InfoFrame. TV detects → disables motion smoothing, noise reduction, upscaling → reduces input lag from 50ms to <10ms. User doesn't need to manually switch to "game mode." |
| **Audio challenge** | Console outputs Dolby Atmos (TrueHD; ~18 Mbps bitstream). Traditional ARC (1 Mbps) can't carry it. eARC (37 Mbps) carries uncompressed Atmos → soundbar receives full object audio. |
| **Cable requirement** | Ultra High Speed HDMI certified cable required for 4K@120Hz 10bpc (needs ~32 Gbps → exceeds HDMI 2.0's 18 Gbps). Many users initially had issues with old "High Speed" cables → certification program is critical. |
| **Compatibility issue** | Many TVs initially advertised "HDMI 2.1" but only supported some features (e.g., 4K@120Hz but no VRR). HDMI Forum doesn't mandate all features → consumer confusion. Solution: check specific feature support (VRR, ALLM, eARC) not just "HDMI 2.1" label. |

### 9.2 8K TV: HDMI 2.1 Bandwidth Challenges

| Aspect | Detail |
|--------|--------|
| **Product** | Samsung 85" 8K QLED TV (7680×4320@60Hz, 10bpc) |
| **Bandwidth** | 8K@60Hz 10bpc RGB: 7680 × 4320 × 30 × 60 = 59.7 Gbps (active pixels). HDMI 2.1 effective bandwidth: ~42.7 Gbps (FRL 4L12). **Problem: 59.7 > 42.7 Gbps.** |
| **Solution 1: YCbCr 4:2:0** | Half chroma resolution: effective BW = 59.7 × 0.667 = ~40 Gbps → fits in 42.7 Gbps. Visually acceptable for video content (human eye less sensitive to chroma). Used for streaming/movies. |
| **Solution 2: DSC** | VESA DSC 1.2a at 2.67:1: 59.7 / 2.67 = ~22.4 Gbps → easily fits. Maintains 10bpc RGB quality (visually lossless). Used for PC connection (where RGB matters). |
| **Solution 3: Wait for HDMI 2.2** | 96 Gbps: supports 8K@60Hz 10bpc RGB uncompressed (59.7 < 85 Gbps effective). True RGB gaming at 8K. |
| **Source devices** | As of 2024: almost no consumer devices output native 8K (GPUs can, but few games render at 8K). Most 8K content is upscaled. HDMI 2.2 will coincide with GPU hardware capable of real 8K gaming. |

## Chapter 11 — Interview Questions & Career Guide

### Tier 1: Entry-Level

**Q1:** What are the main differences between HDMI 2.0 and HDMI 2.1?

**A:**

| Feature | HDMI 2.0 | HDMI 2.1 |
|:-------:|:---:|:---:|
| **Max bandwidth** | 18 Gbps | **48 Gbps** (2.67× more) |
| **Signaling** | TMDS (3 channels + clock) | **FRL** (4 lanes; no separate clock) |
| **Max resolution** | 4K@60Hz (limited to 4:2:0 at 10bpc) | **8K@60Hz; 4K@120Hz** (full RGB 10bpc) |
| **Gaming** | No VRR; no ALLM | **VRR + ALLM** (essential for gaming) |
| **Audio** | ARC (limited; compressed only) | **eARC** (37 Mbps; uncompressed Dolby Atmos) |
| **Compression** | None | **DSC** (enables even higher resolutions) |
| **Error correction** | None | **FEC** (Reed-Solomon; corrects cable errors) |
| **Cable** | High Speed HDMI | **Ultra High Speed HDMI** (certified 48 Gbps) |
| **Dynamic HDR** | No (static HDR10 only) | **Yes** (HDR10+, Dolby Vision frame-by-frame) |

**In summary:** HDMI 2.1 brings 2.67× more bandwidth (enabling 4K@120Hz gaming and 8K), plus gaming features (VRR, ALLM), better audio (eARC for Atmos), and robustness (FEC).

### Tier 2: Mid-Level

**Q2:** Explain Fixed Rate Link (FRL) in HDMI 2.1 — how it differs from TMDS, the encoding scheme, and why it was necessary.

**A:**

**Why TMDS couldn't scale:**
- TMDS uses a separate clock channel (1 of the 4 pairs carries clock only → wastes a differential pair)
- Only 3 data channels; max 6 Gbps each → ceiling of 18 Gbps
- 8b/10b encoding per channel: 20% overhead
- Clock is pixel-clock-derived: scaling to higher rates means impossibly high clock frequencies for the clock channel
- No FEC: at higher speeds, bit errors increase but TMDS has no error correction

**FRL solution:**
- **4 data lanes** (repurposes the old clock pair as a 4th data lane): 33% more data channels
- **Embedded clock** (clock recovered from data transitions, like PCIe/USB): no wasted pair
- **16b/18b encoding**: 11.1% overhead (vs. 20% for 8b/10b) → more efficient
- **FEC per lane**: Reed-Solomon RS(255,251) corrects up to 2 symbol errors per block → tolerates cable imperfections
- **Scalable rates**: FRL levels from 3 Gbps/lane to 12 Gbps/lane (and 24 Gbps in HDMI 2.2)

**16b/18b encoding:**
- Every 16 bits of data → encode as 18 bits on wire
- Overhead: 2/18 = 11.1% (vs. 8b/10b's 2/10 = 20%)
- Includes RS parity bytes for FEC
- Scrambled for DC balance (LFSR scrambler)

**Link Training (FRL):**
- Similar to DisplayPort: source and sink negotiate the highest mutual FRL level
- Link Training Sequence (LTS):
  1. LTS:1 — Pattern generation (source sends; sink checks)
  2. LTS:2 — Rate negotiation (try highest → fall back if errors)
  3. LTS:3 — Normal operation (video data)
- If cable can't support FRL 4L12, fall back to FRL 4L10, 4L8, etc.
- If FRL entirely fails (very old cable), fall back to TMDS mode (HDMI 2.0 compatibility)

### Tier 3: Senior

**Q3:** Design the HDMI subsystem for a next-generation AV receiver that must handle 4 HDMI 2.1 inputs, 1 output (to TV), and eARC. Discuss bandwidth management, HDCP, audio extraction, and latency.

**A:**

**Architecture:**

```
HDMI Input 1 ─┐
HDMI Input 2 ─┤
HDMI Input 3 ─├─→ HDMI Switch Fabric → HDMI TX → TV (via eARC-capable port)
HDMI Input 4 ─┘        ↓
                  Audio Extraction
                        ↓
                  Audio DSP (Dolby Atmos / DTS:X decode)
                        ↓
                  DAC → Amplifiers → 7.1.4 Speaker Array
```

**Component design:**

| Component | Specification | Key Challenges |
|:---------:|:---:|---|
| **HDMI RX × 4** | FRL 4L12 (48 Gbps each) | 4 independent FRL receivers; each with CDR, 16b/18b decode, FEC decode |
| **Switch fabric** | 48 Gbps passthrough (selected input → output) | Zero-latency switching; no re-encoding (bitstream passthrough for video) |
| **HDMI TX × 1** | FRL 4L12 (48 Gbps) to TV | Must support all FRL levels; EDID forwarding from TV to sources |
| **eARC TX** | 37 Mbps (on same HDMI cable to TV) | ARC from TV (for TV apps) → extracted to AVR audio DSP |
| **Audio extraction** | Extract audio from HDMI stream (each input) | Multi-format: PCM 7.1, TrueHD (Atmos), DTS:X, DD+. Low-latency extraction (<1ms). |
| **HDCP** | HDCP 2.3 (all ports) | RX authenticates as sink to each source; TX re-authenticates to TV. Must NOT decrypt/re-encrypt (adds latency). Use HDCP repeater topology. |
| **VRR passthrough** | Pass VRR timing from source to TV | AVR must be transparent to VRR (not buffer frames; not introduce latency) |
| **ALLM passthrough** | Forward ALLM InfoFrame source→TV | Parse InfoFrame; pass through; also trigger AVR's own low-latency mode |

**HDCP Repeater Design:**

| Step | Action |
|:----:|--------|
| 1 | AVR acts as HDCP **repeater** (not converter): authenticates downstream (TV) first |
| 2 | Reports downstream device count + depth to upstream source |
| 3 | Source authenticates with AVR-as-repeater; includes downstream topology |
| 4 | Content flows encrypted end-to-end (AVR does NOT decrypt video; only decrypt audio stream for extraction) |
| 5 | **Key:** AVR must decrypt ONLY the audio portion (HDCP allows this for audio processing) while keeping video encrypted |

**Latency management:**

| Path | Target Latency | Design |
|:----:|:--------------:|--------|
| Video passthrough (input → output) | < 1 ms | No frame buffering; bitstream passthrough via switch fabric |
| Audio extraction | < 2 ms | Extract audio packets from HDMI stream; no video involvement |
| VRR transparency | 0 ms added | No frame buffering means VRR timing passes through unmodified |
| ALLM detection | < 100 ms | Parse AVI InfoFrame on input; forward to output + set AVR mode |
| eARC (TV audio → AVR) | < 5 ms | Dedicated eARC receiver; direct to audio DSP |

**Bandwidth challenge (4 inputs active):**

- Only 1 input is "selected" (passed to TV output) at a time for video
- But audio from all 4 inputs might need monitoring (e.g., multi-zone) → need audio extraction on all 4 simultaneously
- Each input's FRL receiver operates independently (no bandwidth sharing needed)
- Switch fabric: 48 Gbps per input; only routes 1 to output → single 48 Gbps TX path
