ROLE & CONTEXT
--------------------------------------------------------------------------------
Act as a distributed systems architect, RPC framework designer, API platform
engineer, and service-oriented architecture specialist.

RPC (Remote Procedure Call) = a mechanism that allows a program to execute a
procedure in another address space (same machine or remote) as if it were a
local function call, hiding the network or IPC details.

Stack position:
    Schema / Serialization
        ↓
    RPC FRAMEWORKS ← (this volume)
        ↓
    Transport Layer

An RPC framework provides:
  1. An IDL or contract language to define service methods
  2. Code generation to produce client stubs and server skeletons
  3. A serialization layer to encode call arguments and return values
  4. A transport layer to carry encoded messages
  5. A dispatch mechanism on the server side
  6. Error handling, metadata, deadline/cancellation propagation

SECTION 1 — GRPC
================================================================================

1.1  History: Google internal Stubby → gRPC open-sourced 2015, CNCF graduated 2017
1.2  IDL: Protocol Buffers .proto (service + rpc definitions)
1.3  Transport: HTTP/2 (mandatory), QUIC (experimental via gRPC-over-HTTP/3)
1.4  Serialization: Protobuf (default), JSON (grpc-gateway / transcoding), custom codec
1.5  gRPC call types:
       - Unary RPC: single request, single response
       - Server streaming: single request, stream of responses
       - Client streaming: stream of requests, single response
       - Bidirectional streaming: both sides stream simultaneously
1.6  HTTP/2 framing of gRPC:
       - HEADERS frame: :method=POST, :path=/pkg.Service/Method, content-type=application/grpc
       - DATA frames: 5-byte prefix (compression flag + 4-byte message length) + Protobuf bytes
       - TRAILERS frame: grpc-status, grpc-message at stream end
1.7  gRPC metadata: key-value pairs sent as HTTP/2 headers (initial and trailing)
1.8  gRPC status codes: OK, CANCELLED, UNKNOWN, INVALID_ARGUMENT, DEADLINE_EXCEEDED,
       NOT_FOUND, ALREADY_EXISTS, PERMISSION_DENIED, RESOURCE_EXHAUSTED,
       FAILED_PRECONDITION, ABORTED, OUT_OF_RANGE, UNIMPLEMENTED, INTERNAL,
       UNAVAILABLE, DATA_LOSS, UNAUTHENTICATED
1.9  Deadlines and cancellation: propagated as HTTP/2 grpc-timeout header
1.10 Interceptors: client-side and server-side middleware chain
1.11 Service reflection: grpc.reflection.v1alpha.ServerReflection for runtime introspection
1.12 gRPC health checking: grpc.health.v1.Health.Check standard protocol
1.13 gRPC-Web: browser-compatible subset (no HTTP/2 trailer streaming in browsers)
1.14 gRPC-Gateway: transcodes REST+JSON ↔ gRPC+Protobuf
1.15 gRPC-Binder: gRPC transport over Android Binder (same-device IPC)
1.16 Language support: C++, Java, Go, Python, Rust, C#, Node.js, Swift, Kotlin, PHP
1.17 Governance: Google / CNCF (Tier 6/5)

ANNOTATED FLOW:
  Client code calls stub.GetVehicleSpeed(request)
    → stub serializes request with Protobuf
    → 5-byte length prefix added
    → HTTP/2 HEADERS frame sent (method, path, content-type)
    → HTTP/2 DATA frame sent (length-prefix + proto bytes)
    → Server receives DATA frame
    → Server strips 5-byte prefix
    → Server deserializes Protobuf bytes into request message
    → Server handler executes business logic
    → Server serializes response with Protobuf
    → Server sends HTTP/2 DATA frame (response) + TRAILERS frame (grpc-status=0)
    → Client receives and deserializes response

SECTION 2 — APACHE THRIFT RPC
================================================================================

2.1  History: Facebook 2007 → Apache 2008
2.2  IDL: Thrift IDL (.thrift files with service definitions)
2.3  Architecture: TTransport + TProtocol + TServer (layered design)
2.4  TTransport options: TSocket, TServerSocket, TFramedTransport, TBufferedTransport,
       THttpTransport, TMemoryBuffer, TZlibTransport
2.5  TProtocol options: TBinaryProtocol, TCompactProtocol, TJSONProtocol,
       TSimpleJSONProtocol, TMultiplexedProtocol
2.6  TServer options: TSimpleServer, TThreadedServer, TThreadPoolServer, TNonblockingServer
2.7  Async vs sync clients
2.8  Exception handling: Thrift exceptions (IDL-defined) vs TTransportException vs TApplicationException
2.9  Service multiplexing over one connection: TMultiplexedProcessor
2.10 Oneway methods: fire-and-forget (no response expected)
2.11 Language support: C++, Java, Python, PHP, Ruby, Go, Haskell, Perl, Node.js, C#
2.12 Governance: Apache Software Foundation (Tier 5)

================================================================================
SECTION 3 — JSON-RPC
================================================================================

3.1  Specification: JSON-RPC 1.0 (2005), JSON-RPC 2.0 (2010) — Tier 8
3.2  Transport agnostic: HTTP, WebSockets, TCP, stdio, named pipes
3.3  Request object: jsonrpc="2.0", method, params (array or object), id
3.4  Response object: jsonrpc="2.0", result XOR error, id
3.5  Error object: code (integer), message, data (optional)
3.6  Notification: request without id (no response expected)
3.7  Batch requests: array of request objects
3.8  Standard error codes: -32700=Parse error, -32600=Invalid Request,
       -32601=Method not found, -32602=Invalid params, -32603=Internal error
3.9  No IDL: method names and params are strings/objects — no code generation
3.10 Usage: Ethereum JSON-RPC, LSP (Language Server Protocol), Neovim RPC,
              bitcoin-core, Electrum, web3 libraries

================================================================================
SECTION 4 — XML-RPC
================================================================================

4.1  Specification: xmlrpc.com (1998) — Tier 8
4.2  Transport: HTTP POST
4.3  Request: XML body with <methodCall><methodName><params>
4.4  Response: XML body with <methodResponse><params> or <fault>
4.5  Types: int, boolean, string, double, dateTime.iso8601, base64, struct, array
4.6  Status: largely superseded by JSON-RPC and REST; still in some legacy systems
4.7  Historical significance: precursor to SOAP

================================================================================
SECTION 5 — SOAP (Simple Object Access Protocol)
================================================================================

5.1  Specifications: SOAP 1.1 (W3C Note, 2000), SOAP 1.2 (W3C Recommendation, 2003)
5.2  Transport: HTTP (primary), SMTP, JMS, others (transport binding defined in WSDL)
5.3  Message structure:
       <soap:Envelope>
         <soap:Header> (optional, for auth/routing/WS-* extensions)
         <soap:Body> (required, the actual call and response)
       </soap:Envelope>
5.4  SOAP 1.1: Content-Type: text/xml
     SOAP 1.2: Content-Type: application/soap+xml
5.5  WSDL: the service contract language for SOAP services
5.6  WS-* stack: WS-Security, WS-Addressing, WS-AtomicTransaction, WS-ReliableMessaging
5.7  MTOM (Message Transmission Optimization Mechanism): binary data in SOAP without base64
5.8  Fault structure: <soap:Fault><faultcode><faultstring><detail>
5.9  Status: actively used in enterprise integration, banking, insurance, healthcare (HL7)
5.10 Governance: W3C (Tier 2)

================================================================================
SECTION 6 — REST (Representational State Transfer)
================================================================================

6.1  REST is an architectural style, NOT a protocol (Fielding dissertation, 2000)
6.2  REST constraints:
       - Client-server separation
       - Statelessness
       - Cacheability
       - Uniform interface (resource-based, self-descriptive, HATEOAS)
       - Layered system
       - Code on demand (optional)
6.3  REST over HTTP: GET, POST, PUT, PATCH, DELETE, HEAD, OPTIONS
6.4  HTTP status codes as error signaling (200, 201, 204, 400, 401, 403, 404, 409, 422, 500)
6.5  Content negotiation: Accept, Content-Type headers
6.6  REST is NOT an IDL/RPC system — OpenAPI provides the contract layer
6.7  RESTful maturity model (Richardson):
       Level 0 = HTTP as a tunnel (one URI, one method)
       Level 1 = Resources (multiple URIs)
       Level 2 = HTTP verbs (GET/POST/PUT/DELETE semantic use)
       Level 3 = HATEOAS (hypermedia controls in responses)
6.8  Common payload: JSON (most common), XML, YAML, Protobuf (Google APIs)

================================================================================
SECTION 7 — CORBA RPC
================================================================================

7.1  History: OMG CORBA 1.1 (1991), CORBA 3.0 (2002)
7.2  IDL: OMG IDL (interface + operation definitions)
7.3  Object Request Broker (ORB): the middleware runtime
7.4  GIOP (General Inter-ORB Protocol) + IIOP (Internet Inter-ORB Protocol over TCP)
7.5  CDR (Common Data Representation): CORBA's serialization format
7.6  Portable Object Adapter (POA): server-side object lifecycle management
7.7  Naming Service, Trading Service, Event Service: standard CORBA services
7.8  Status: largely replaced by gRPC, REST, and messaging systems in new development
7.9  Legacy presence: telecom (CORBA in SS7 / TMN interfaces), defense, embedded
7.10 Governance: OMG (Tier 3)

================================================================================
SECTION 8 — JAVA RMI
================================================================================

8.1  Java Remote Method Invocation: Java-only RPC
8.2  Interface: extends java.rmi.Remote, methods throw RemoteException
8.3  Stubs: generated by rmic compiler (modern JVMs generate dynamically)
8.4  Transport: JRMP (Java Remote Method Protocol) over TCP
8.5  Serialization: Java Object Serialization (not portable to other languages)
8.6  RMI-IIOP: variant running over CORBA IIOP transport for interop
8.7  Limitations: Java-only, Java serialization vulnerabilities (deserialization attacks)
8.8  Status: legacy; replaced by gRPC, REST in new Java services

================================================================================
SECTION 9 — MICROSOFT WCF / .NET REMOTING
================================================================================

9.1  .NET Remoting: legacy .NET 1.x remote object framework (deprecated)
9.2  WCF (Windows Communication Foundation):
       - Binding: defines transport + encoding (BasicHttpBinding, NetTcpBinding,
                  WSHttpBinding, NetNamedPipeBinding, NetMsmqBinding)
       - Contract: [ServiceContract], [OperationContract], [DataContract], [MessageContract]
       - Serialization: DataContractSerializer (default), XmlSerializer, NetDataContractSerializer
       - Security: WS-Security, transport-level TLS
9.3  WCF → gRPC migration path (Microsoft-provided tooling)
9.4  Governance: Microsoft (Tier 6/7)

================================================================================
SECTION 10 — CAP'N PROTO RPC
================================================================================

10.1 Capability-based RPC system (not just an encoding format)
10.2 Interface definitions in .capnp files (interface keyword)
10.3 Promise pipelining: send follow-up calls on not-yet-received results
10.4 Object capabilities: passing interface references between processes
10.5 Transport: TCP, Unix Domain Sockets, custom
10.6 Language support: C++, Rust (capnproto-rust), Python, JavaScript
10.7 Governance: Sandstorm.io / open-source (Tier 8)

================================================================================
SECTION 11 — APACHE DUBBO
================================================================================

11.1 Alibaba internal → Apache (2018)
11.2 IDL: Protobuf .proto (gRPC-compatible mode) or Dubbo-native interface annotations
11.3 Protocols: Dubbo (binary TCP), Triple (HTTP/2, gRPC-compatible), REST
11.4 Service registry: Zookeeper, Nacos, Consul, etcd
11.5 Load balancing: Random, RoundRobin, LeastActive, ConsistentHash
11.6 Cluster fault tolerance: Failover, Failfast, Failsafe, Broadcast
11.7 Governance: Apache Software Foundation (Tier 5)
11.8 Usage: large-scale Chinese internet companies (Alibaba ecosystem)

================================================================================
SECTION 12 — SUN RPC / ONC-RPC
================================================================================

12.1 Origin: Sun Microsystems; Open Network Computing RPC
12.2 Specification: RFC 5531 (ONC RPC)
12.3 IDL: XDR (External Data Representation) — RFC 4506
12.4 XDR: 4-byte aligned, big-endian, fixed-size arrays + variable-size with length prefix
12.5 Port mapping: portmapper / rpcbind (well-known port 111)
12.6 Usage: NFS (Network File System), NIS, rpc.mountd — core Unix infrastructure
12.7 Governance: IETF (Tier 2) — RFC 5531, RFC 4506
