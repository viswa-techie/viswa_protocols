ROLE & CONTEXT
--------------------------------------------------------------------------------
Act as a standards policy analyst, technology governance expert, open-source
foundation specialist, and intellectual property strategist.

This volume maps every major standards body, foundation, and vendor involved
in the governance of data contracts, serialization, IDLs, API contracts, and
interoperability technologies.

Key principle: A technology is only as stable as its governance model.
Understanding who controls a standard determines: longevity risk, fork risk,
patent encumbrance, open-source licensing, and implementation diversity.

SECTION 1 — GOVERNANCE TIER LADDER (FULL DEFINITION)
================================================================================

TIER 0 — MATHEMATICAL / FORMAL FOUNDATIONS
  Examples : Formal grammars (BNF, EBNF), type theory, algebraic data types,
              lambda calculus, set theory for data models
  Authority: No governing body — mathematical truth
  Stability : Permanent
  Relevance : Foundational basis for all formal schema and IDL languages

TIER 1 — INTERNATIONAL TREATY-LEVEL STANDARDS BODIES
  Bodies    : ISO, IEC, ISO/IEC JTC1
  Authority : Intergovernmental treaty; national standards body members
  Process   : Multi-year consensus process; national body voting
  Examples  :
    ISO 8824 / ITU-T X.680 (ASN.1)
    ISO/IEC 9075 (SQL)
    ISO 20022 (Financial messaging)
    IEC 61508 (Functional safety)
    ISO 26262 (Automotive safety)
    IEC 62541 (OPC UA)
    ISO/IEC 19505 (UML — jointly with OMG)
  Patents   : FRAND licensing for essential patents required for ISO standards
  Stability : Very high; backward-incompatible changes extremely rare

TIER 2 — INTERGOVERNMENTAL / GLOBAL INTERNET STANDARDS BODIES
  Bodies    : ITU-T, IETF, IEEE, W3C
  Authority : Global consensus; major implementers drive adoption
  Process   :
    IETF   : RFC process (Individual → Working Group → IESG → RFC Editor)
    ITU-T  : ITU-T Recommendations (X., G., H., Q. series); national member bodies
    W3C    : Working Draft → Candidate Rec → Proposed Rec → Recommendation
    IEEE   : Standards project → ballot → approval
  Examples  :
    IETF: RFC 8259 (JSON), RFC 8949 (CBOR), RFC 7950 (YANG), RFC 6241 (NETCONF),
           RFC 7252 (CoAP), RFC 7519 (JWT), RFC 5531 (ONC-RPC), RFC 4506 (XDR)
    ITU-T: X.680 (ASN.1), X.690 (BER/DER), X.691 (PER), X.693 (XER), X.509 (PKI)
    W3C  : XML 1.0, XSD 1.1, SOAP 1.2, WSDL 2.0, SPARQL, RDF, OWL
    IEEE : IEEE 754 (floating point), IEEE 802.x (networking)
  Stability : High; formal IP disclosure required

TIER 3 — INDUSTRY CONSORTIA STANDARDS
  Bodies    : OMG, AUTOSAR, SAE, 3GPP, COVESA, OpenAPI Initiative, OPC Foundation,
               OASIS, W3C TAG, GraphQL Foundation (Linux Foundation sub-project)
  Authority : Consortium member companies vote; membership-based
  Process   : Member proposals → working group → specification → ratification
  Examples  :
    OMG       : CORBA IDL, DDS, DDS-XTypes, UML, MOF
    AUTOSAR   : AUTOSAR Classic/Adaptive specs, SOME/IP, ARXML
    3GPP      : LTE/5G ASN.1 specs (TS 38.331, TS 38.413, etc.)
    OpenAPI Initiative : OpenAPI Specification 3.x
    OASIS     : SAML, OData, AMQP 1.0, WS-BPEL
    COVESA    : Vehicle Signal Specification, Franca IDL
    OPC Foundation : OPC UA (IEC 62541)
    SAE       : Automotive protocols, AUTOSAR liaison
    ETSI      : NFV data models, MEC APIs
  Stability : Medium-high; members have commercial interests

TIER 4 — PROFESSIONAL / TRADE ASSOCIATIONS
  Bodies    : FIX Trading Community, HL7 International, SWIFT, FIXML TC,
               Object Management Group (overlaps Tier 3)
  Authority : Industry practitioners; membership from financial/healthcare firms
  Examples  :
    FIX Trading : FIX Protocol, SBE, FIXML
    HL7         : HL7 v2, HL7 v3, FHIR
    SWIFT       : SWIFT MT, MX (ISO 20022 implementation)
  Stability : Medium; driven by industry adoption

TIER 5 — OPEN-SOURCE FOUNDATION STANDARDS
  Bodies    : Apache Software Foundation, Linux Foundation, CNCF, Eclipse Foundation,
               Open Robotics Foundation, Python Software Foundation, OpenJS Foundation
  Authority : Committer community + foundation board; open governance
  Examples  :
    Apache ASF      : Avro, Thrift, Kafka, Arrow, Dubbo
    Linux Foundation: Zephyr RTOS, Iceoryx, ACRN, Yocto, OpenEmbedded
    CNCF            : gRPC (graduated), Protobuf (sandbox), CloudEvents, AsyncAPI (proposed)
    Eclipse Foundation : Iceoryx, Eclipse Cyclone DDS, Eclipse Kuksa
    Open Robotics   : ROS, ROS2
    GraphQL Foundation (LF) : GraphQL specification
  Stability : Medium; community can fork; governance through TSC/PMC

TIER 6 — VENDOR-LED OPEN SOURCE (Single vendor primary control)
  Examples  :
    Google    : Protocol Buffers, gRPC, FlatBuffers, Flatbuffers2, Abseil, FIDL
    Amazon    : Smithy IDL, AWS CDK (CloudFormation schema)
    Microsoft : Bond IDL, TypeSpec (Cadl), WCF
    Meta      : Apache Thrift (originated), Hack/HHVM protocols
    HashiCorp : HCL (HashiCorp Configuration Language)
    Confluent : Schema Registry, ksqlDB schema handling
    MongoDB   : BSON specification
  Stability : Vendor risk: if vendor discontinues, community may fork or abandon
  License   : Typically Apache 2.0 or MIT; patents via CLA

TIER 7 — VENDOR PROPRIETARY (Closed specification)
  Examples  : Internal Google formats not published, vendor-specific wire formats
               in closed automotive ECU software, SWIFT proprietary extensions
  Stability : Controlled by vendor; black box to external implementers

TIER 8 — DE-FACTO STANDARDS (No formal body)
  Examples  :
    JSON        : De-facto standard (formalized later as ECMA-404 and RFC 8259)
    REST        : Architectural style from Fielding dissertation, no formal body
    YAML        : Community specification at yaml.org
    MessagePack : Community specification at msgpack.org
    Cap'n Proto : Informal specification
    JSON-RPC    : Community specification
    GraphQL (pre-LF) : Facebook internal → de-facto → GraphQL Foundation
  Stability : Variable; depends on ecosystem adoption momentum

SECTION 7 — VENDOR GOVERNANCE ANALYSIS
================================================================================

7.1  Google
     Technologies: Protocol Buffers, gRPC, FlatBuffers, FlatBuffers2, Abseil, FIDL
     License: Apache 2.0 / BSD 3-Clause
     CLA: Google CLA required
     Risk: Google has deprecated products before (Google+, Cloud IoT Core)
     Mitigation: gRPC donated to CNCF (reduces single-vendor dependency)

7.2  Amazon Web Services
     Technologies: Smithy IDL, CloudFormation schema, Event Bridge schema registry
     License: Apache 2.0 (Smithy)
     Risk: AWS-centric toolchain; cross-cloud portability concerns

7.3  Microsoft
     Technologies: Bond IDL, TypeSpec (Cadl), WCF, MIDL
     License: MIT (Bond, TypeSpec)
     Mitigation: TypeSpec designed to emit OpenAPI (reduces lock-in)

7.4  Apache Software Foundation
     Governance: PMC model; any contributor can become committer
     IP: Apache License 2.0; software grants required
     Projects: Avro (PMC: Apache Avro), Thrift (PMC: Apache Thrift), Arrow (PMC: Apache Arrow)

7.5  Confluent
     Technologies: Confluent Schema Registry (originally open source, then BSL)
     License change risk: Confluent changed Schema Registry to BSL 1.1 in 2023
     Alternatives: Apicurio Registry (Red Hat / open-source), AWS Glue Schema Registry
