transmitted. Data modeling is the FIRST layer in the universal stack:

    Application Objects → DATA MODELING ← (this volume)

                        IDL / Contract
                            ↓
                        Schema
                            ↓
                        Serialization

--------------------------------------------------------------------------------
Cover every major paradigm, language, notation, and standard used to DEFINE
data structure, relationships, semantics, and constraints at the conceptual,
logical, and physical modeling levels.

SECTION 2 — RELATIONAL DATA MODELING
================================================================================

2.1  Entity-Relationship (ER) model — Chen notation (1976)
     - Entities, attributes, relationships, cardinality
     - Strong vs weak entities
     - ISA hierarchies

2.2  Extended ER (EER) model
     - Generalization, specialization, aggregation

2.3  Relational model (Codd, 1970)
     - Relations, tuples, domains, keys
     - First, Second, Third Normal Form
     - BCNF, 4NF, 5NF
     - Functional dependencies, multivalued dependencies

2.4  Unified Modeling Language (UML) Class Diagrams as data models
     - Class, attribute, association, aggregation, composition, generalization
     - Multiplicity, role names, navigability
     - OCL constraints in UML

2.5  Governance: ISO/IEC 9075 (SQL), ISO/IEC 19505 (UML)

SECTION 3 — OBJECT-ORIENTED DATA MODELING
================================================================================

3.1  Object model concepts
     - Identity, state, behavior
     - Encapsulation, inheritance, polymorphism
     - Value types vs reference types

3.2  UML as an OO data modeling standard
     - Class diagram, object diagram, component diagram
     - Packages, namespaces, stereotypes

3.3  Domain-Driven Design (DDD) data modeling
     - Entity vs Value Object vs Aggregate
     - Bounded context and the ubiquitous language
     - Repository and domain event patterns
     - Aggregate root and consistency boundaries

3.4  ORM data models (Hibernate/JPA, SQLAlchemy)
     - How OO models map to relational tables
     - Impedance mismatch problem

================================================================================
SECTION 4 — SEMANTIC & ONTOLOGICAL DATA MODELING
================================================================================

4.1  RDF (Resource Description Framework)
     - Subject–Predicate–Object triple model
     - Blank nodes, literals, IRIs
     - Governance: W3C, RDF 1.1 specification

4.2  OWL (Web Ontology Language)
     - Classes, properties, individuals
     - OWL DL, OWL Full, OWL 2
     - SWRL rules, SPARQL queries
     - Governance: W3C

4.3  RDFS (RDF Schema)
     - rdfs:Class, rdfs:Property, rdfs:subClassOf
     - Inferencing and reasoning

4.4  SKOS (Simple Knowledge Organization System)
     - Concept schemes, broader/narrower/related terms
     - Thesaurus and taxonomy representation

4.5  JSON-LD (JSON for Linked Data)
     - @context, @type, @id
     - How JSON-LD maps to RDF

4.6  Governance: W3C Semantic Web standards stack

================================================================================
SECTION 5 — NETWORK & TELECOM DATA MODELS
================================================================================

5.1  YANG (Yet Another Next Generation — RFC 7950)
     - module, container, leaf, list, leaf-list, choice, grouping
     - Deviation, augment, feature, identity
     - Governance: IETF (RFC 7950, 6020)
     - Bindings: NETCONF (RFC 6241), RESTCONF (RFC 8040)

5.2  SMIv2 (Structure of Management Information v2)
     - MIB modules, OBJECT-TYPE, SEQUENCE, MODULE-IDENTITY
     - Governance: IETF
     - Usage: SNMP MIBs

5.3  CIM (Common Information Model)
     - Managed Object Format (MOF) syntax
     - WMI / WBEM usage on Windows and Linux
     - Governance: DMTF (Distributed Management Task Force)

5.4  OpenConfig models
     - Vendor-neutral network device models
     - gNMI as the transport binding
     - Governance: OpenConfig Working Group

================================================================================
SECTION 6 — INDUSTRY-SPECIFIC DATA MODELS
================================================================================

6.1  FHIR (Fast Healthcare Interoperability Resources)
     - Resource types: Patient, Observation, Condition, Medication
     - FHIR profiles and extensions
     - FHIR JSON, XML, and RDF representations
     - Governance: HL7 International

6.2  HL7 v2 / v3 message models
     - Message segments, fields, components
     - HL7 v3 RIM (Reference Information Model)

6.3  ISO 20022 (Financial messaging)
     - Business Process Catalogue
     - Logical Data Dictionary
     - Message Definition Reports
     - XML Schema output
     - Governance: ISO TC68/SC9

6.4  AUTOSAR data types
     - Primitive types, composite types, application data types
     - Software Component port interfaces
     - Governance: AUTOSAR consortium

6.5  COVESA VSS (Vehicle Signal Specification)
     - Signal tree structure (Vehicle.Speed, Vehicle.Body.*)
     - VSS grammar: branches, sensors, actuators, attributes
     - Governance: COVESA / Eclipse Foundation

6.6  ROS2 message types (.msg / .srv / .action)
     - Field types, constants, arrays
     - IDL-based rosidl code generation
     - Governance: Open Robotics Foundation

================================================================================

7.1  OpenAPI Schema Object as a data model
     - Properties, required, type, format, $ref, allOf/oneOf/anyOf
     - How OpenAPI schemas model domain objects for API consumers

7.2  JSON Schema as a standalone data modeling language
     - type, properties, required, additionalProperties
     - $defs, $ref, $anchor, unevaluatedProperties

7.3  GraphQL Type System
     - Scalar, Object, Interface, Union, Enum, Input types

     - How GraphQL types model a domain graph

7.4  Protobuf message as a data model
     - message, field, oneof, map, repeated
     - Well-known types (Timestamp, Duration, Struct, Value)
     - How Protobuf messages can serve as cross-language data models

SECTION 8 — DATA CONTRACT PATTERNS & PRINCIPLES
================================================================================

8.1  Contract-first vs code-first development

     - Code-first: code written first, schema extracted

     - Consumer contract (what I require to receive)

8.3  Data lineage and provenance modeling
     - How data models track origin and transformation

8.4  Temporal data modeling
     - Bitemporal models (transaction time vs valid time)
     - Event-sourcing data models
     - CQRS and the separation of read/write models
