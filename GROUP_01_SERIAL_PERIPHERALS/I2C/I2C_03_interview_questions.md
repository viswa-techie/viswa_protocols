# PART A: JUNIOR / ENTRY-LEVEL (25 Questions)

---

### Q1. What is I2C and what does it stand for?
**A:** I2C stands for Inter-Integrated Circuit. It's a synchronous, half-duplex, multi-master serial bus invented by Philips (now NXP) in 1982. It uses only 2 wires (SDA + SCL) to connect multiple devices on a shared bus, with each device having a unique address.

---

### Q2. How many wires does I2C use? Name them.
**A:** Two signal wires plus ground:
- **SDA** (Serial Data) — bidirectional data line
- **SCL** (Serial Clock) — clock generated by master
- **GND** — shared ground reference

---

### Q3. What is the role of pull-up resistors in I2C?
**A:** I2C uses open-drain outputs, meaning devices can only pull lines LOW. Pull-up resistors connected to VDD provide the HIGH state when no device is actively pulling LOW. Without pull-ups, the bus would never go HIGH and communication is impossible.

---

### Q4. What is the typical pull-up resistor value for standard mode?
**A:** 4.7KΩ is the standard recommendation for 100 kHz with moderate bus capacitance (~200pF). The actual value depends on VDD, bus capacitance, and speed mode (lower resistance for faster speeds).

---

### Q5. Explain START and STOP conditions.
**A:**
- **START**: SDA transitions HIGH→LOW while SCL is HIGH. Signals beginning of transaction.
- **STOP**: SDA transitions LOW→HIGH while SCL is HIGH. Signals end of transaction.
These are the ONLY times SDA changes while SCL is HIGH (during data transfer, SDA is stable when SCL is HIGH).

---

### Q6. How does addressing work in I2C?
**A:** After START, the master sends a 7-bit slave address plus 1 R/W̄ bit. The slave with matching address responds with ACK (pulls SDA LOW on 9th clock). This eliminates the need for chip-select pins like SPI.

---

### Q7. What is ACK and NACK?
**A:**
- **ACK**: Receiver pulls SDA LOW during the 9th clock pulse → "byte received successfully"
- **NACK**: Receiver leaves SDA HIGH during 9th clock → "problem" or "no more data needed"

---

### Q8. How many devices can you connect to a standard I2C bus?
**A:** With 7-bit addressing: 128 addresses total, minus 16 reserved = **112 usable addresses**. Practical limit is also constrained by bus capacitance (400pF max in standard/fast mode), typically 20-30 devices.

---

### Q9. What are the standard I2C speed modes?
**A:**
- Standard mode (Sm): 100 kHz
- Fast mode (Fm): 400 kHz
- Fast-mode Plus (Fm+): 1 MHz
- High-Speed (HS): 3.4 MHz
- Ultra-Fast (UFm): 5 MHz (push-pull, unidirectional only)

---

### Q10. What's the difference between I2C and SPI?
**A:**

| Feature | I2C | SPI |
|---------|-----|-----|
| Wires | 2 (SDA, SCL) | 4+ (MOSI, MISO, SCK, CS) |
| Addressing | Software (7/10-bit) | Hardware (CS pin per device) |
| Duplex | Half | Full |
| Speed | Up to 3.4 MHz | Up to 200+ MHz |
| Multi-master | Yes | No (typically) |
| Error detection | ACK/NACK | None built-in |
| Pin count | Low (2+N×0) | High (3+N CS) |

---

### Q11. What is the R/W bit in the address byte?
**A:** The LSB of the first byte after START. 0 = Write (master→slave), 1 = Read (slave→master). Example: Address 0x68 → Write=0xD0, Read=0xD1.

---

### Q12. Why is I2C called "half-duplex"?
**A:** There's only one data line (SDA) shared by master and slave. Data can flow only one direction at a time — either master writes or master reads, never both simultaneously.

---

### Q13. What happens if no slave responds to an address?
**A:** No device pulls SDA LOW during the ACK clock → master reads NACK. Master should abort the transaction with STOP. In Linux, this returns -ENXIO or -EREMOTEIO.

---

### Q14. What is a Repeated START? Why is it needed?
**A:** A START condition issued without a preceding STOP. Used for combined write-then-read transactions (e.g., write register address then read data). It keeps the bus locked so another master can't intervene between the write and read phases.

---

### Q15. What is TWI?
**A:** TWI (Two-Wire Interface) is Atmel's name for I2C. They used a different name to avoid Philips' trademark (before the patent expired). Electrically and protocol-wise, it's identical to I2C.

---

### Q16. What does `i2cdetect` do?
**A:** A Linux utility that probes all possible addresses on an I2C bus and shows which ones respond with ACK. Output is a grid showing detected device addresses. Example: `i2cdetect -y 0` scans bus 0.

---

### Q17. What voltage levels does I2C support?
**A:** I2C works at 1.8V, 2.5V, 3.3V, and 5V. Devices at different voltages need level shifters. The open-drain nature makes level shifting relatively simple using a MOSFET.

---

### Q18. Can you have multiple masters on I2C?
**A:** Yes! I2C supports multi-master. When two masters try to communicate simultaneously, arbitration (via wired-AND) determines the winner. The loser detects it lost and backs off.

---

### Q19. What is the bus capacitance limit?
**A:** 400 pF maximum for standard and fast mode. This includes all device input capacitance (~5-10pF each), PCB trace capacitance (~1pF/cm), and connector capacitance.

---

### Q20. Name 5 common I2C devices.
**A:**
1. EEPROM (AT24C256) — data storage
2. Temperature sensor (LM75/TMP112) — thermal monitoring
3. RTC (DS3231) — timekeeping
4. IMU/Accelerometer (MPU6050) — motion sensing
5. GPIO Expander (PCF8574) — more I/O pins

---

### Q21. What are reserved I2C addresses?
**A:** 0x00 (general call), 0x01 (CBUS), 0x02-0x03 (reserved), 0x04-0x07 (HS master code), 0x78-0x7B (10-bit prefix), 0x7C-0x7F (reserved/Device ID). That's 16 addresses unavailable for normal slaves.

---

### Q22. In what order are bits transmitted?
**A:** MSB first. Bit 7 of each byte is transmitted first, bit 0 last. After 8 data bits, the 9th clock is for ACK/NACK.

---

### Q23. What is a "general call" in I2C?
**A:** Address 0x00 with Write bit = broadcast to all devices. Used for software reset (second byte 0x06), address programming, or latching commands. Only devices that support general call respond.

---

### Q24. How does I2C differ from UART?
**A:**

| Feature | I2C | UART |
|---------|-----|------|
| Clock | Synchronous (SCL) | Asynchronous (baud rate) |
| Topology | Multi-device bus | Point-to-point |
| Addressing | Yes (7/10-bit) | No |
| Wires | 2 | 2 (TX/RX) |
| Speed | Up to 3.4 MHz | Up to 5 Mbaud typically |
| Duplex | Half | Full |

---

### Q25. What is open-drain and why does I2C use it?
**A:** Open-drain means the output can only actively pull LOW (MOSFET connects to GND). It can't drive HIGH — the pull-up resistor does that. I2C uses this because: (1) enables wired-AND for multi-master arbitration, (2) prevents bus damage when two devices drive simultaneously, (3) simplifies level shifting between different voltages.

# PART B: MID-LEVEL (30 Questions)

---

### Q26. How do you calculate the correct pull-up resistor value?
**A:**
- **Rp_min** = (VDD - V_OL) / I_OL = (3.3 - 0.4) / 3mA ≈ 967Ω → 1KΩ minimum
- **Rp_max** = t_rise / (0.8473 × C_bus)
  - Fast mode (300ns rise): Rp_max = 300ns / (0.8473 × 200pF) = 1.77KΩ
  - Standard (1µs rise): Rp_max = 1µs / (0.8473 × 200pF) = 5.9KΩ

If Rp too high: slow rise time, bus fails at target speed. If too low: sink current exceeds device spec.

---

### Q27. Explain I2C arbitration in detail.
**A:** When multiple masters start simultaneously:
1. Both drive SDA while monitoring it (open-drain → wired-AND)
2. Bit-by-bit comparison: if a master drives HIGH but reads LOW, it means another master is driving LOW → it lost
3. The loser immediately stops driving and waits for STOP before retrying
4. The winner continues unaware that arbitration occurred
5. No data corruption because wired-AND means the winning data is what's on the bus
6. Lower addresses have priority (more zeros early = more likely to win)

---

### Q28. What is clock stretching and when is it problematic?
**A:** Clock stretching is when a slave holds SCL LOW to pause the master while processing. The master must check SCL is actually HIGH before proceeding.

**Problems:**
- No timeout in I2C spec → slave can hang bus forever
- Some hardware masters don't support clock stretching (ignore SCL readback)
- Bit-bang implementations often forget to check
- SMBus adds 35ms timeout to prevent stuck buses

---

### Q29. How do you perform a register read over I2C?
**A:** Two-phase transaction with Repeated START:
1. **Write phase**: S → [Addr+W] → ACK → [Reg] → ACK
2. **Read phase**: Sr → [Addr+R] → ACK → [Data] → NACK → P

The Repeated START (Sr) is crucial — it keeps the bus locked between setting the register pointer (write) and reading the data (read). Using STOP+START instead would release the bus, allowing another master to intercept.

---

### Q30. What causes an I2C bus to get "stuck"? How do you recover?
**A:** **Causes:** Slave holding SDA LOW (crashed mid-byte, power glitch reset master but not slave, clock glitch).

**Recovery (9-clock pulse method):**
1. Configure SCL as GPIO output, SDA as GPIO input
2. Toggle SCL up to 9 times
3. After each clock, check if SDA released (HIGH)
4. Once SDA is HIGH, generate STOP condition (SDA LOW→HIGH while SCL HIGH)
5. If SDA still stuck after 9 clocks → hardware reset needed

---

### Q31. How does I2C level shifting work with a MOSFET?
**A:** Using N-channel MOSFET (BSS138) with gate tied to lower voltage:
- **Low-side pulls LOW**: MOSFET source pulled low → Vgs > Vth → drain pulled low → both sides go LOW
- **High-side pulls LOW**: Drain pulled low → body diode conducts → source pulled low → both sides LOW
- **Both release**: Pull-ups on each side bring lines to their respective VDD levels
- Bidirectional, automatic, simple — that's why I2C level shifting is easier than SPI.

---

### Q32. What's the difference between I2C and SMBus?
**A:**

| Feature | I2C | SMBus |
|---------|-----|-------|
| Min clock | DC (can stop indefinitely) | 10 kHz (can't pause >35ms) |
| Max clock | 3.4 MHz | 100 kHz (1.x) / 1 MHz (3.x) |
| Timeout | None! | 35ms (prevents stuck bus) |
| Error check | ACK only | PEC: CRC-8 polynomial |
| Electrical | Flexible | Tighter specs |
| Alert | None | SMBALERT# interrupt line |

---

### Q33. Explain 10-bit addressing.
**A:** First byte: 11110 + A9:A8 + R/W. Second byte: A7:A0. Provides 1024 addresses. 7-bit devices and 10-bit devices coexist because the 11110 prefix is reserved from 7-bit address space. Rarely used in practice.

---

### Q34. How do you handle I2C in a multi-threaded/RTOS environment?
**A:** Use a mutex/semaphore to protect the I2C bus:
```c
mutex_lock(&i2c_bus_mutex);
i2c_write_reg(addr, reg, val);
mutex_unlock(&i2c_bus_mutex);
```
Without this, two threads could interleave transactions (Thread A's START, Thread B's address → corrupted). The mutex must protect the entire compound transaction (including Repeated START sequences).

---

### Q35. What Linux tools and interfaces exist for I2C debugging?
**A:**
- `i2cdetect -y N` — scan bus, find devices
- `i2cget -y N addr reg` — read register
- `i2cset -y N addr reg val` — write register
- `i2cdump -y N addr` — dump all registers
- `i2ctransfer -y N w1@addr reg r2` — raw transfer
- `/sys/bus/i2c/devices/` — sysfs entries
- `/dev/i2c-N` — userspace access via ioctl
- `dmesg | grep i2c` — kernel messages
- Logic analyzer (Saleae, PulseView) — waveform capture

### Q36. How do you configure I2C in a Linux Device Tree?
**A:**
```dts
&i2c2 {
    status = "okay";
    clock-frequency = <400000>;  /* Fast mode */
    pinctrl-names = "default";
    pinctrl-0 = <&i2c2_pins>;
    
    sensor@48 {
        compatible = "ti,tmp112";
        reg = <0x48>;
    };
};
```
Key properties: `clock-frequency`, `reg` (slave address), `compatible` (driver binding), `status` = "okay" enables the bus.

---

### Q37. What is the I2C General Call address and its uses?
**A:** Address 0x00 with W bit. Broadcasts to all slaves. Second byte defines command:
- 0x06: Software reset — all devices reset
- 0x04: Write programming address
- Vendor-specific commands

Not all devices support general call — it's optional per spec.

---

### Q38. How do you read multiple bytes from a device?
**A:** After the Repeated START and address+Read, the master sends ACK after each byte (meaning "more please") and NACK only on the last byte (meaning "stop sending"). Most devices auto-increment their register pointer, so consecutive reads return consecutive registers.

---

### Q39. Explain the I2C peripheral clock configuration on STM32.
**A:**
```
Standard mode (100 kHz): CCR = APB_CLK / (2 × 100000)
Fast mode (400 kHz): CCR = APB_CLK / (3 × 400000)  [2:1 duty]
TRISE = (APB_MHz × t_rise_ns / 1000) + 1
  Standard: TRISE = (42 × 1000/1000) + 1 = 43
  Fast: TRISE = (42 × 300/1000) + 1 = 13
```
The I2C peripheral clock must be derived from APB1 and the APB frequency must be at least 2 MHz (standard) or 4 MHz (fast).

---

### Q40. What is an I2C multiplexer and when would you use one?
**A:** A device (e.g., PCA9548A) that routes one I2C bus to multiple downstream channels. Use when:
1. Address conflict (two devices with same address on different channels)
2. Bus capacitance exceeded (split into shorter segments)
3. Fault isolation (one segment failure doesn't kill entire bus)

Control by writing channel select mask to the mux's own address.

---

### Q41. What are common causes of I2C NAK errors in Linux?
**A:**
1. Wrong address (7-bit vs 8-bit confusion)
2. Device not powered or in reset
3. Device not assembled on board
4. Wrong I2C bus (device on i2c-1, code uses i2c-0)
5. Pull-ups missing or wrong value
6. Voltage level mismatch
7. Device in sleep/shutdown mode
8. Device address pins not configured correctly

---

### Q42. How does DMA work with I2C?
**A:** DMA handles data transfer between I2C peripheral data register and memory buffer:
- Master configures: slave address, register, byte count, buffer pointer
- DMA transfers each byte automatically (DR→buffer for reads, buffer→DR for writes)
- CPU freed during transfer, interrupt on completion
- Critical for multi-byte transfers (e.g., reading 6 bytes from accelerometer at high rate)

---

### Q43. What is I2C bus capacitance and why does it matter?
**A:** Sum of all capacitance on SDA/SCL: device pins (~5-10pF each), PCB traces (~1pF/cm), connectors (~2-5pF). Total must be <400pF (standard/fast). Higher capacitance → slower rise time → can't achieve target frequency. Solutions: stronger pull-ups, active pull-ups, bus buffers, or I2C mux to segment.

---

### Q44. What happens if you use internal MCU pull-ups instead of external?
**A:** Internal pull-ups are typically 20-50KΩ — way too weak for I2C. Rise time will be extremely slow. Might work at very low speed (<50 kHz) with minimal bus capacitance, but unreliable. **Always use external pull-ups** for production designs. Internal pull-ups are acceptable only for prototyping at low speed.

---

### Q45. How do you detect clock stretching as a master?
**A:** After releasing SCL HIGH, the master reads back SCL:
- If SCL is actually HIGH → proceed
- If SCL is still LOW → slave is stretching → wait until SCL goes HIGH

Hardware I2C peripherals typically handle this automatically. Bit-bang implementations must explicitly check.

---

### Q46. What is SCL frequency vs. duty cycle?
**A:** At 400 kHz (fast mode):
- Period = 2.5 µs
- t_LOW ≥ 1.3 µs (mandatory minimum)
- t_HIGH ≥ 0.6 µs (mandatory minimum)
- Duty cycle: not 50/50! LOW must be longer than HIGH
- Common: 2:1 ratio (t_LOW : t_HIGH = 1.67:0.83 µs)

---

### Q47. Explain the `i2c_msg` structure in Linux kernel.
**A:**
```c
struct i2c_msg {
    __u16 addr;    /* 7-bit slave address */
    __u16 flags;   /* I2C_M_RD, I2C_M_TEN, I2C_M_NOSTART, etc. */
    __u16 len;     /* message length (bytes) */
    __u8 *buf;     /* data buffer */
};
```
Used by `i2c_transfer()`. A register read requires 2 msgs: write msg (register addr) + read msg (data). The framework handles Repeated START between them.

---

### Q48. What is PEC in SMBus?
**A:** Packet Error Checking — an 8-bit CRC appended after the last data byte. Polynomial: x⁸ + x² + x¹ + 1 (CRC-8). Covers address, read/write bit, and all data bytes. Provides error detection that standard I2C lacks.

---

### Q49. How does I2C compare to I3C?
**A:**

| Feature | I2C | I3C |
|---------|-----|-----|
| Speed | Up to 3.4 MHz | 12.5 MHz (HDR) |
| Signaling | Open-drain only | Push-pull + open-drain |
| Interrupts | Separate GPIO | In-Band Interrupt (IBI) |
| Addressing | Static | Dynamic assignment |
| Hot-join | No | Yes |
| Backward compat | — | Supports legacy I2C devices |

---

### Q50. What's the difference between `i2c_smbus_read_byte_data()` and `i2c_transfer()`?
**A:**
- `i2c_smbus_read_byte_data(client, reg)` — high-level SMBus API, reads one byte from register. Handles Repeated START internally.
- `i2c_transfer(adap, msgs, num)` — low-level, sends raw i2c_msg array. More flexible (multi-byte, non-standard sequences) but more complex.

Use SMBus API for standard register access; use i2c_transfer for complex/custom protocols.

---

### Q51. How do you handle EEPROM page writes over I2C?
**A:** EEPROMs have page boundaries (typically 32 or 64 bytes). A write that crosses a page boundary wraps around within the page — data is corrupted! You must:
1. Break writes at page boundaries
2. Wait for write cycle (5-10ms) after each page — poll ACK or delay
3. Use ACK polling: send address repeatedly until device ACKs (write complete)

---

### Q52. What is "byte banging" (bit-banging) I2C?
**A:** Implementing I2C in software using GPIO pins instead of hardware peripheral. Toggle SCL GPIO, read/write SDA GPIO with proper timing. Advantages: works on any GPIOs, supports clock stretching easily. Disadvantages: CPU-intensive, timing-sensitive, slower. Linux: `i2c-gpio` driver.

---

### Q53. How does the Linux i2c-gpio (bit-bang) driver work?
**A:** Device tree specifies GPIO pins for SDA/SCL:
```dts
i2c-gpio {
    compatible = "i2c-gpio";
    sda-gpios = <&gpio 4 GPIO_OPEN_DRAIN>;
    scl-gpios = <&gpio 5 GPIO_OPEN_DRAIN>;
    i2c-gpio,delay-us = <2>;  /* ~250 kHz */
};
```
The algo-bit.c module implements I2C protocol using GPIO toggle. Supports clock stretching by reading SCL GPIO state.

---

### Q54. What's the I2C subsystem registration flow in Linux?
**A:**
1. Platform driver probe → allocate `struct i2c_adapter`
2. Set `adapter->algo` (master_xfer function pointer)
3. Call `i2c_add_adapter()` or `i2c_add_numbered_adapter()`
4. I2C core creates `/dev/i2c-N` and scans Device Tree
5. For each child node → create `i2c_client`, bind to matching driver
6. Client driver's `probe()` called with client pointer

---

### Q55. How do you write a basic I2C client driver in Linux?
**A:**
```c
static int my_sensor_probe(struct i2c_client *client) {
    u8 id = i2c_smbus_read_byte_data(client, WHO_AM_I_REG);
    if (id != EXPECTED_ID) return -ENODEV;
    /* Register with subsystem (hwmon, input, iio, etc.) */
    return 0;
}

static const struct of_device_id my_ids[] = {
    { .compatible = "vendor,my-sensor" },
    { }
};

static struct i2c_driver my_driver = {
    .driver = { .name = "my-sensor", .of_match_table = my_ids },
    .probe = my_sensor_probe,
};
module_i2c_driver(my_driver);
```

### Q56. Design an I2C subsystem for an automotive ECU with 15 sensors.
**A:**
1. **Bus architecture**: Use 2-3 I2C buses + PCA9548A mux for isolation
   - Bus 0: Critical (PMIC, RTC) — Standard mode, short traces
   - Bus 1: Sensors (temp, IMU, ADC) — Fast mode
   - Bus 2 with mux: Expansion (IO expanders, LED drivers)
2. **Pull-ups**: Calculate per-bus based on device count and trace length
3. **Error handling**: Bus recovery mechanism (9-clock), watchdog on transactions
4. **Power domains**: Level shifters where buses cross voltage domains
5. **RTOS integration**: Per-bus mutex, priority inheritance to prevent inversion
6. **Diagnostics**: Transaction counters, NACK counters, bus recovery events
7. **Redundancy**: Critical sensors on separate buses for fail-safety

---

### Q57. How would you debug intermittent I2C NACK errors?
**A:** Systematic approach:
1. **Logic analyzer capture** — look for timing violations, glitches, noise
2. **Check rise/fall times** — measure with scope vs. spec requirements
3. **Bus capacitance** — measure with impedance meter, verify <400pF
4. **Pull-up verification** — measure actual resistance, check for parallel pull-ups
5. **EMI investigation** — does it correlate with motor/relay switching?
6. **Power supply** — brownouts causing device reset during transaction
7. **Temperature** — some devices have temp-dependent timing
8. **Software race** — multiple threads accessing bus without proper locking
9. **Clock stretching** — slave stretching beyond master's timeout
10. **Addressing** — check for address conflicts with hidden devices (mux, buffer)

### Q58. Explain I2C bus recovery in Linux kernel.
**A:**
```c
struct i2c_bus_recovery_info {
    int (*recover_bus)(struct i2c_adapter *adap);
    int (*get_scl)(struct i2c_adapter *adap);
    void (*set_scl)(struct i2c_adapter *adap, int val);
    int (*get_sda)(struct i2c_adapter *adap);
    void (*prepare_recovery)(struct i2c_adapter *adap);
    void (*unprepare_recovery)(struct i2c_adapter *adap);
    struct gpio_desc *scl_gpiod;
    struct gpio_desc *sda_gpiod;
};
```
The framework: (1) switch pins to GPIO mode, (2) toggle SCL up to 9 times checking SDA, (3) generate STOP, (4) switch back to I2C AF mode. Drivers set `adap->bus_recovery_info` and kernel calls `i2c_recover_bus()` on errors.

---

### Q59. How does Qualcomm's QUP I2C controller differ from generic controllers?
**A:** QUP (Qualcomm Universal Peripheral) is a configurable serial engine that can be programmed as I2C, SPI, or UART:
- **QUP v1** (legacy): Register-based programming, FIFO mode, BAM DMA
- **GENI** (QUP v3, newer): Generic Interface, programmed via firmware (SE), GPI DMA
- Features: Configurable FIFO depth, noise rejection, configurable stretch timeout, packed mode (multiple bytes per FIFO word), DMA via GPI engine
- Linux drivers: `i2c-qup.c` (legacy), `i2c-qcom-geni.c` (GENI)
- Automotive (SA8155P/SA8295P): Up to 12 QUP instances, shared with SPI/UART

---

### Q60. What are the timing considerations for I2C at 1 MHz (Fm+)?
**A:**
- Rise time max: 120ns (very aggressive!)
- Pull-ups: ~1KΩ or active pull-ups needed
- Bus capacitance: <550pF (relaxed from 400pF but traces must be shorter)
- I_OL increased to 20mA (vs 3mA standard) — stronger drivers
- PCB layout critical: minimize stubs, keep traces short (<10cm)
- Fewer devices per bus (each adds capacitance)
- Output driver must be Fm+ capable (higher current)
- Check ALL devices support Fm+ — one slow device pulls down entire bus

### Q61. How do you implement I2C device hot-plug in Linux?
**A:** Standard I2C doesn't support hot-plug, but Linux provides mechanisms:
```c
/* Runtime device addition */
struct i2c_board_info info = { I2C_BOARD_INFO("sensor", 0x48) };
struct i2c_client *client = i2c_new_client_device(adapter, &info);

/* Removal */
i2c_unregister_device(client);

/* Sysfs interface */
echo "sensor 0x48" > /sys/bus/i2c/devices/i2c-0/new_device
echo "0x48" > /sys/bus/i2c/devices/i2c-0/delete_device
```
For true hot-plug: detect via interrupt/GPIO, then add device. Must handle bus capacitance changes.

### Q62. Design a fault-tolerant I2C system for safety-critical automotive use.
**A:**
1. **Bus segmentation**: PCA9548A mux isolates segments; fault in one doesn't kill others
2. **Redundancy**: Dual I2C buses for critical sensors (compare readings)
3. **Watchdog**: Transaction timeout with bus recovery
4. **Diagnostics**: Track NACK count, bus stuck events, CRC mismatches
5. **Protection**: TVS diodes for ESD, series resistors for overcurrent
6. **SMBus timeout**: Implement 35ms timeout even on I2C to prevent hangs
7. **Pull-up monitoring**: Detect open/short on SDA/SCL lines
8. **Safe state**: Define behavior when I2C communication lost (failsafe values)
9. **AUTOSAR**: DEM (Diagnostic Event Manager) reporting for I2C faults

---

### Q63. Explain I2C repeated START vs STOP+START: when does it matter?
**A:** In single-master systems, they're often equivalent. But they differ when:
1. **Multi-master**: Sr keeps bus locked (arbitration won't occur between phases). STOP+START releases bus → another master can grab it → you lose atomic read-modify-write
2. **Register pointer integrity**: Some devices reset register pointer on STOP. Repeated START preserves it
3. **EEPROM sequential read**: Must use Sr to maintain read position
4. **SMBus block read**: Protocol requires Sr between command and data phases

**Always use Repeated START for register reads** unless you're certain single-master and device retains pointer after STOP.

### Q64. How do I2C buffers/repeaters work?
**A:** I2C buffers (e.g., PCA9515, P82B715) regenerate I2C signals:
- Extend bus beyond 400pF capacitance limit
- Each side has its own pull-ups (capacitance split)
- Buffer senses LOW on one side → drives LOW on other side
- Some are bidirectional with dead zone detection
- P82B715 also does voltage translation and cable extension (>10m)

Limitations: Can't use across multi-master buses (arbitration breaks), add propagation delay.

### Q65. What is clock synchronization in I2C multi-master?
**A:** When multiple masters drive SCL simultaneously:
- SCL goes LOW when ANY master pulls it LOW (wired-AND)
- SCL goes HIGH only when ALL masters release it
- This means: fastest master waits for slowest master to release SCL
- Natural synchronization — no explicit protocol needed
- Combined with clock stretching by slave → all masters automatically synchronized to slowest participant

### Q66. How do you test I2C implementation for EMC compliance?
**A:**
1. **Emissions**: Measure RF emissions from I2C lines (harmonics of clock)
2. **Immunity**: Inject EFT/ESD/surge per IEC 61000-4-x, verify no communication loss
3. **Bulk Current Injection**: Inject RF current onto I2C cables
4. **Mitigation**: 
   - Ferrite beads on I2C lines (careful: adds inductance)
   - Capacitive filtering (careful: adds bus capacitance)
   - Shielded cables for external I2C
   - Reduce clock frequency if emissions high
   - Series resistors (33-100Ω) to reduce edge rate

### Q68. What is I2C "NAK storm" and how to prevent it?
**A:** When a driver continuously polls a non-existent device, generating NAK on every attempt. This floods the bus with useless traffic and wastes time.

Prevention:
- Check device presence once at probe, fail fast
- Exponential backoff on NAK
- Limit retry count (Linux i2c core has `adap->retries`)
- Use device-ready GPIO/interrupt instead of polling
- In Linux: `i2c_check_functionality()` and proper error handling

### Q70. Explain I2C in power management — wake/sleep scenarios.
**A:**
- **Before sleep**: Configure wake-capable I2C devices (e.g., PMIC, touch)
- **During sleep**: Bus idle. Wake sources (PMIC interrupt via GPIO) trigger resume
- **Resume path**: I2C controller re-initialized, pull-ups still powered, re-probe devices
- **Power islands**: Some devices powered off during sleep → I2C bus sees capacitance change → re-enumerate at wake
- **Linux PM callbacks**: `suspend()` / `resume()` in i2c_driver restore device config
- **I2C wakeup**: NOT standard — use separate GPIO interrupt, not I2C bus activity

### Q76. How does I2C interact with device power states (D0/D3)?
**A:**
- **D0 (active)**: Device fully responsive on I2C
- **D3 (off)**: Device may not respond; might pull SDA LOW during power transition!
- **Issues**: Device powering up/down can glitch I2C bus → other devices affected
- **Solutions**: 
  - I2C mux to isolate power domains
  - Power sequencing: ensure I2C devices powered before master starts scanning
  - Bus recovery after power domain transitions
  - Design: tie device SDA/SCL to power rail through weak pull-ups that drop when device depowered

### Q77. What considerations for I2C in mixed-signal (analog+digital) designs?
**A:**
1. **Ground loops**: I2C return current through shared ground → noise injection into ADCs
2. **Crosstalk**: SCL harmonics couple into analog signals → route I2C away from analog
3. **Pull-up noise**: Switching current in pull-ups couples to VDD → add decoupling
4. **Clock radiation**: SCL is periodic → strong EMI at fundamental and harmonics
5. **Mitigation**: Separate I2C ground plane, ferrites on I2C to analog devices, reduce SCL frequency for ADC buses, read ADC during I2C idle periods

### Q78. How would you implement I2C over a cable (>1m)?
**A:**
1. **Bus buffer** (P82B715): Amplifies signals, drives cable capacitance
2. **Reduced speed**: 10-100 kHz max for longer cables
3. **Twisted pair**: SDA with GND, SCL with GND (minimize loop area)
4. **Active termination**: Bus buffers at each end
5. **Differential**: Convert to differential (e.g., PCA9600) for noise immunity
6. **Shielding**: Shielded cable for EMC
7. **Alternative**: Convert to RS-485 for long distance, use I2C-to-RS485 bridge
8. **Practical limit**: ~3-5m with buffers, 10m+ with differential conversion

### Q79. Explain I2C device reset mechanisms.
**A:**
1. **Hardware reset**: Dedicated reset pin on device — toggle GPIO
2. **Software reset**: General call (0x00) + 0x06 command → all devices reset
3. **Bus reset**: 9 clock pulses + STOP → unstick hung devices
4. **Power cycle**: Cut VDD to device via load switch
5. **Peripheral reset**: Reset I2C controller on MCU (CR1.SWRST on STM32)
6. **Nuclear option**: Full system reset

Priority: try soft reset → bus reset → hard reset → power cycle → system reset.

# PART E: DEBUG SCENARIOS (5 Real-World Problems)

---

### Scenario 1: "i2cdetect shows no devices"

**Symptoms:** `i2cdetect -y 0` returns all dashes (--). No device found.

**Debug Steps:**
1. Verify hardware:
   - Are pull-ups physically present? Measure R between SDA/SCL and VDD
   - Are devices powered? Measure VDD at device power pins
   - Are devices soldered correctly? Check for solder bridges
2. Verify GPIO configuration:
   - `cat /sys/kernel/debug/pinctrl/*/pins` — check SDA/SCL muxed correctly
   - GPIO must be in AF (alternate function) mode, not regular GPIO
   - Verify open-drain + pull-up configuration
3. Verify I2C controller:
   - `dmesg | grep i2c` — check driver probe succeeded
   - `ls /dev/i2c-*` — check device nodes exist
   - Is the controller clock enabled? (RCC/clock gate)
4. Scope check:
   - Put scope on SDA/SCL — do you see START condition?
   - If no SCL toggle → master not generating clocks → controller not initialized
   - If SCL toggles but SDA stays LOW → bus stuck → recovery needed

**Root Cause (common):** Missing pull-ups, wrong GPIO config, or controller clock not enabled.

---

### Scenario 2: "Random data corruption every ~1000 reads"

**Symptoms:** Sensor readings occasionally wrong. No NACK, device responds, but data garbled.

**Debug Steps:**
1. **Scope**: Capture corrupted transaction — look for:
   - Slow rise time (SDA barely reaching VIH)
   - Glitches on SCL (extra clock pulses → bit shift)
   - Noise on SDA during clock HIGH time → wrong bit sampled
2. **Pull-up**: Rise time too slow for speed → increase pull-up (lower resistance)
3. **EMI**: Correlate corruption with motor/relay/display activity
4. **Power**: Check VDD for dips during high-current events
5. **Timing**: SDA setup/hold time margin — may fail at temperature extremes
6. **Software**: Race condition in multi-threaded access without mutex

**Root Cause (common):** Marginal rise time + EMI induced glitch that occasionally corrupts one bit.

---

### Scenario 3: "Bus locks up after running for 2 hours"

**Symptoms:** I2C transactions start failing (timeout). Bus doesn't recover after reboot of software. Only power cycle fixes it.

**Debug Steps:**
1. **Scope on SDA/SCL** when stuck:
   - SDA stuck LOW → slave holding it → clock recovery needed
   - SCL stuck LOW → clock stretching slave hung
   - Both HIGH → bus actually free, master driver problem
2. **Identify stuck slave**: Remove devices one by one (mux helps isolate)
3. **Check**: Is bus recovery implemented? Try 9 clock pulses.
4. **Root cause hunting**:
   - Power glitch reset master mid-transaction → slave waiting for clock
   - Interrupt handler taking too long → slave's clock stretch exceeded → state lost
   - Noise on SCL → slave miscounted clocks → expects more clocks → holds SDA
5. **Fix**: Implement hardware watchdog + automatic bus recovery. Add bus stuck detection (poll bus state, if stuck >100ms → recovery procedure). Consider hardware reset GPIO to slave as nuclear option.

---

### Scenario 4: "Device works on bench but not in vehicle"

**Symptoms:** I2C sensor works perfectly on development board. Fails (NACK/garbage) in actual vehicle ECU.

**Debug Steps:**
1. **EMI**: Vehicle is electrically noisy (CAN, motors, ignition)
   - Add ferrites on I2C lines
   - Add filtering capacitors (carefully, within bus cap budget)
   - Check ground impedance between devices
2. **Temperature**: ECU near engine → operating at 85°C+ → timing margins shrink
3. **Voltage**: Vehicle power supply fluctuates (cranking = 6V dip) → regulator droop
4. **Vibration**: Connector intermittent → measure continuity under vibration
5. **Cable length**: Vehicle wiring longer than bench → capacitance budget exceeded
6. **Ground offset**: Different ground references between ECU and sensor → voltage level shifted

**Root Cause (common):** EMI from vehicle environment + longer cable capacitance exceeding bus spec.

---

### Scenario 5: "Touch controller stops responding after display update"

**Symptoms:** Touch (0x5D) and display driver (0x3C) on same I2C bus. After display firmware update, touch stops responding.

**Debug Steps:**
1. **Address conflict**: Display update changed device address? Scan bus before/after
2. **Bus busy**: Display FW update holds bus for long write sequences → touch times out
3. **Clock stretching**: Display in bootloader stretches clock excessively
4. **Power rail**: Display update causes power fluctuation affecting touch
5. **Bus stuck**: Display bootloader crashed mid-transaction → SDA stuck
6. **Solution**:
   - Put display and touch on separate I2C buses (best)
   - Schedule: disable touch polling during display update
   - Bus recovery after display update completes
   - Separate power domain with controlled sequencing
